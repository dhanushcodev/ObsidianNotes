# 🔐 Spring Security Authentication Case Sensitivity (Your Example Explained)

## 🚨 Observation from Your Output

```java
principal.getUsername() → dhanush
loginRequestDto.username() → Dhanush
```

👉 Input = `"Dhanush"`  
👉 Stored = `"dhanush"`  
👉 Authentication still succeeds ✅

---

## 🤔 Why is authentication successful?

You are using:

```java
return new InMemoryUserDetailsManager(user, admin);
```

### 🔍 Internal Behavior

`InMemoryUserDetailsManager` stores usernames in a **case-insensitive way**.

Simplified internal logic:

```java
users.put(username.toLowerCase(), userDetails);
```

During authentication:

```java
inputUsername.toLowerCase()
```

So:

```java
"Dhanush".toLowerCase() → "dhanush"
```

👉 That’s why authentication succeeds.

---

## 🧠 Important Understanding

### ✔ Works like this:

- InMemoryUserDetailsManager → case-insensitive usernames

### ❌ Does NOT work like this:

- JPA / Hibernate (Database)
- Custom UserDetailsService

Example:

```java
findByUsername("Dhanush")
```

If DB has `"dhanush"` → ❌ FAIL (case mismatch)

---

## 🔑 Password Behavior (Very Important)

Password check:

```java
passwordEncoder().matches(rawPassword, encodedPassword);
```

Examples:

```java
"1234"  → ✅ matches
"12345" → ❌ fails
```

👉 Password is ALWAYS **case-sensitive** (for security reasons)

---

## ⚠️ Real-World Warning

This behavior will break when you switch to database-based authentication.

You must handle case-insensitivity manually:

### Option 1: Convert to lowercase

```java
username = username.toLowerCase();
```

### Option 2: Case-insensitive query

```java
@Query("SELECT u FROM User u WHERE LOWER(u.username) = LOWER(:username)")
User findByUsernameIgnoreCase(String username);
```

---

## 🧾 Final Summary

|Field|Case Sensitive?|Behavior|
|---|---|---|
|Username|❌ (here)|Case-insensitive due to InMemoryUserDetailsManager|
|Username|✅ (DB)|Case-sensitive unless handled manually|
|Password|✅ ALWAYS|Case-sensitive (security requirement)|

---

## ✅ Conclusion

Your authentication works because:

- InMemoryUserDetailsManager converts usernames to lowercase internally
- Password matches correctly
- Username case difference is ignored in this specific setup

👉 Do NOT rely on this behavior when moving to production (DB-based auth)
