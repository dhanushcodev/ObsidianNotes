# Spring Boot Validation (Complete Notes – Basics + All Mappings)

---

## 🔹 1. Overview

Validation in Spring Boot ensures that incoming request data is correct before processing.

- Uses **Jakarta Bean Validation (JSR-380)**
- Implemented by **Hibernate Validator**
- Helps:
	- Prevent invalid data
    - Reduce manual checks
    - Improve API reliability

---

## 🔹 2. Dependency

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-validation</artifactId>
</dependency>
```

## 🔹 3. Common Validation Annotations

|Annotation|Description|
|---|---|
|`@NotNull`|Must not be null|
|`@NotEmpty`|Not null + not empty|
|`@NotBlank`|Not null + trimmed not empty|
|`@Size`|String length constraint|
|`@Email`|Valid email|
|`@Min / @Max`|Number range|
|`@Pattern`|Regex validation|
|`@Past / @Future`|Date validation|

## 🔹 4. DTO Example with Validation

```java
import jakarta.validation.constraints.*;

public class UserDTO {

    @NotBlank(message = "Name is required")
    private String name;

    @Email(message = "Invalid email format")
    private String email;

    @Min(value = 18, message = "Age must be >= 18")
    private int age;

    @Size(min = 6, max = 12, message = "Password must be 6-12 chars")
    private String password;

    // getters & setters
}
```

## 🔹 5. Validation Based on Mapping Types

---

### ✅ `@PostMapping` (Create)

- Used for request body validation
    

```java
@PostMapping
public String createUser(@Valid @RequestBody UserDTO user) {
    return "User created";
}
```

✔ Uses `@Valid`  
✔ Triggers DTO validation

---

### ✅ `@PutMapping` (Update)

```java
@PutMapping("/{id}")
public String updateUser(
        @PathVariable int id,
        @Valid @RequestBody UserDTO user) {

    return "User updated";
}
```

✔ Same as POST

---

### ✅ `@GetMapping` (Query Params Validation)

- Requires `@Validated` at class level
    

```java
@RestController
@Validated
@RequestMapping("/users")
public class UserController {

    @GetMapping
    public String getUsers(
            @RequestParam @Min(18) int age,
            @RequestParam @NotBlank String name) {

        return "Valid request";
    }
}
```

---

### ✅ `@GetMapping` (Path Variable)

```java
@GetMapping("/{id}")
public String getUserById(@PathVariable @Min(1) int id) {
    return "Valid ID";
}
```

---

### ✅ `@DeleteMapping`

```java
@DeleteMapping("/{id}")
public String deleteUser(@PathVariable @Min(1) int id) {
    return "User deleted";
}
```

✔ Uses `@Validated`

## 🔹 7. Exception Types

|Scenario|Exception|
|---|---|
|POST / PUT|`MethodArgumentNotValidException`|
|GET / DELETE|`ConstraintViolationException`|

---

## 🔹 8. Global Exception Handling

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public Map<String, String> handleValidationErrors(MethodArgumentNotValidException ex) {

        Map<String, String> errors = new HashMap<>();

        ex.getBindingResult().getFieldErrors().forEach(error -> {
            errors.put(error.getField(), error.getDefaultMessage());
        });

        return errors;
    }

    @ExceptionHandler(ConstraintViolationException.class)
    public Map<String, String> handleConstraintViolation(ConstraintViolationException ex) {

        Map<String, String> errors = new HashMap<>();

        ex.getConstraintViolations().forEach(error -> {
            errors.put(error.getPropertyPath().toString(), error.getMessage());
        });

        return errors;
    }
}
```

## 🔹 9. `@Valid` vs `@Validated`

|Feature|`@Valid`|`@Validated`|
|---|---|---|
|Source|Jakarta|Spring|
|Use Case|Request Body|Method-level validation|
|Supports Groups|❌|✅|

---

## 🔹 10. Validation Flow

1. Client sends request
    
2. Spring converts JSON → DTO
    
3. `@Valid` / `@Validated` triggers validation
    
4. If invalid → Exception thrown
    
5. Global handler catches it
    
6. Error response returned

## 🔹 11. Custom Validation (Advanced)

### Step 1: Create Annotation

```java
@Constraint(validatedBy = PasswordValidator.class)
@Target({ ElementType.FIELD })
@Retention(RetentionPolicy.RUNTIME)
public @interface ValidPassword {
    String message() default "Invalid password";
    Class<?>[] groups() default {};
    Class<? extends Payload>[] payload() default {};
}
```

### Step 2: Validator Class

```java
public class PasswordValidator implements ConstraintValidator<ValidPassword, String> {

    @Override
    public boolean isValid(String value, ConstraintValidatorContext context) {
        return value != null && value.matches(".*[A-Z].*");
    }
}
```

### Step 3: Usage

```java
@ValidPassword
private String password;
```

## 🔹 12. Summary

- `@PostMapping / @PutMapping`
    
    - Use `@Valid` with `@RequestBody`
        
- `@GetMapping / @DeleteMapping`
    
    - Use `@Validated` for query params and path variables
        
- DTO-based validation is best practice
    
- Handle errors using `@RestControllerAdvice`
    

---

## 🔥 Interview Tip

👉 “Use `@Valid` for request body and `@Validated` for query/path parameters.  
POST/PUT throws `MethodArgumentNotValidException`,  
GET/DELETE throws `ConstraintViolationException`.”
