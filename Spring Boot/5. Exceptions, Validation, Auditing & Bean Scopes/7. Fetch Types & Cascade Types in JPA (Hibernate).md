
## 🔹 1. Fetch Types in JPA

Fetch type defines **WHEN related entities are loaded from the database**.

### ✅ Types of Fetching

#### 1. EAGER Fetching

- Data is loaded **immediately along with parent entity**
- Default for:
    - `@ManyToOne`
    - `@OneToOne`

#### 2. LAZY Fetching

- Data is loaded **only when explicitly accessed**
- Default for:
    - `@OneToMany`
    - `@ManyToMany`

---

### 📌 Example: FetchType

```java
@Entity
class Department {

    @Id
    @GeneratedValue
    private Long id;

    private String name;

    @OneToMany(mappedBy = "department", fetch = FetchType.LAZY)
    private List<Employee> employees;
}

@Entity
class Employee {

    @Id
    @GeneratedValue
    private Long id;

    private String name;

    @ManyToOne(fetch = FetchType.EAGER)
    private Department department;
}
```

---

### 🔍 Behavior

#### LAZY

```java
Department dept = departmentRepository.findById(1L).get();

// Employees NOT loaded yet
dept.getEmployees(); // 🔥 DB query triggered here
```

#### EAGER

```java
Employee emp = employeeRepository.findById(1L).get();

// Department already loaded
emp.getDepartment(); // ❌ No extra query
```

---

### ⚠️ Problems

#### 🔴 EAGER Issues

- Loads unnecessary data
- Can cause **N+1 query problem**
- Performance degradation

#### 🔴 LAZY Issues

- `LazyInitializationException` if session is closed

---

### ✅ Best Practices

- Prefer **LAZY** for collections
- Use **JOIN FETCH / EntityGraph** when needed

## 🔹 2. Cascade Types in JPA

Cascade defines **WHAT operations should propagate from parent to child entity**.

---

### ✅ Cascade Types

|Type|Description|
|---|---|
|`PERSIST`|Save child when parent is saved|
|`MERGE`|Update child when parent is updated|
|`REMOVE`|Delete child when parent is deleted|
|`REFRESH`|Refresh child when parent is refreshed|
|`DETACH`|Detach child when parent is detached|
|`ALL`|Applies all above|

---

### 📌 Example: Cascade

```java
@Entity
class Order {

    @Id
    @GeneratedValue
    private Long id;

    private String orderName;

    @OneToMany(mappedBy = "order", cascade = CascadeType.ALL)
    private List<Item> items;
}

@Entity
class Item {

    @Id
    @GeneratedValue
    private Long id;

    private String itemName;

    @ManyToOne
    private Order order;
}
```

---

### 🔍 Behavior

#### ✅ Cascade Persist

```java
Order order = new Order();
order.setOrderName("Food Order");

Item item1 = new Item();
item1.setItemName("Pizza");

Item item2 = new Item();
item2.setItemName("Burger");

order.setItems(List.of(item1, item2));

orderRepository.save(order);
```

👉 Automatically saves:

- Order
- Item1
- Item2

---

#### ❌ Without Cascade

```java
orderRepository.save(order);
```

👉 Error:

```
object references an unsaved transient instance
```

---

#### ✅ Cascade Remove

```java
orderRepository.delete(order);
```

👉 Deletes:

- Order
- All associated Items

---

## 🔥 Combined Example (Fetch + Cascade)

```java
@Entity
class Blog {

    @Id
    @GeneratedValue
    private Long id;

    private String title;

    @OneToMany(
        mappedBy = "blog",
        fetch = FetchType.LAZY,
        cascade = CascadeType.ALL
    )
    private List<Post> posts;
}

@Entity
class Post {

    @Id
    @GeneratedValue
    private Long id;

    private String content;

    @ManyToOne(fetch = FetchType.EAGER)
    private Blog blog;
}
```

## 🧠 Understanding Together

|Concept|Meaning|
|---|---|
|FetchType|**WHEN data is loaded**|
|CascadeType|**WHAT operations propagate**|

---

## ⚡ Interview Tips

### 🔹 Key Differences

|Feature|FetchType|CascadeType|
|---|---|---|
|Purpose|Data loading|Operation propagation|
|Concern|Performance|Data consistency|
|Example|LAZY / EAGER|ALL / PERSIST / REMOVE|

---

### 🔹 Best Practices

- Use `FetchType.LAZY` by default
- Avoid `CascadeType.ALL` blindly
- Use cascade only when:
    - Child lifecycle depends on parent

---

## 🚀 Real-World Analogy

### 🔹 FetchType

- Like loading comments of a post:
    - LAZY → Load when user clicks
    - EAGER → Load immediately

### 🔹 CascadeType

- Like deleting a folder:
    - Delete folder → delete all files inside
