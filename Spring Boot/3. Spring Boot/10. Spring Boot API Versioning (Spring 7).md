# Spring Boot API Versioning (Spring 7)

## 📌 Overview
Spring Framework 7 (and Spring Boot 3.3+) provides **first-class API versioning support**.

👉 Versioning is:
- Centrally configured
- Type-safe
- Integrated into request mapping

---

## ⚙️ Configuration (Central Setup)

```java
@Configuration
public class WebConfig implements WebMvcConfigurer {

    @Override
    public void configureApiVersioning(ApiVersionConfigurer configurer) {
        configurer
            .usePathSegment(2)   // ZERO-based index
            .addSupportedVersions("1.0", "2.0");
    }
}
```

---

## 🔢 Understanding `usePathSegment()`

👉 Uses ZERO-based indexing

### Example:

```
/api/versions/1.0/users
```

| Index | Segment   |
|------|----------|
| 0    | api      |
| 1    | versions |
| 2    | 1.0 ✅   |
| 3    | users    |

✔ `usePathSegment(2)` → extracts `"1.0"`

---

## 🔍 Query Parameter Versioning (`useQueryParam()`)

### Configuration:

```java
configurer
    .useQueryParam("version")
    .addSupportedVersions("1.0", "2.0");
```

### Example Requests:

```
GET /api/hello?version=1.0 → v1()
GET /api/hello?version=2.0 → v2()
```

✔ Version is extracted from query parameter

---

## 📩 Header Versioning (`useRequestHeader()`)

Version can also be passed via HTTP headers.

### Configuration:

```java
configurer
    .useRequestHeader("X-API-Version")
    .addSupportedVersions("1.0", "2.0");
```

### Example Requests:

```
GET /api/hello
Header: X-API-Version: 1.0 → v1()

GET /api/hello
Header: X-API-Version: 2.0 → v2()
```

✔ Clean URLs  
✔ Common in enterprise APIs  

---

## 📦 Media Type Versioning (`useMediaTypeParameter()`)

Version is passed as part of the **Accept header**.

### Configuration:

```java
configurer
    .useMediaTypeParameter("v")
    .addSupportedVersions("1.0", "2.0");
```

### Example Requests:

```
GET /api/hello
Accept: application/json;v=1.0 → v1()

GET /api/hello
Accept: application/json;v=2.0 → v2()
```

✔ Follows REST content negotiation  
✔ Used in advanced API designs  

---

## 🎯 Default Version (`setDefaultVersion()`)

### Configuration:

```java
configurer
    .useQueryParam("version")
    .setDefaultVersion("1.0")
    .addSupportedVersions("1.0", "2.0");
```

### Behavior:

```
GET /api/hello              → v1() (default)
GET /api/hello?version=2.0  → v2()
```

✔ Prevents errors when version is missing  

---

## 🔁 Switching Versioning Strategy (🔥 Easy in Spring 7)

👉 The **big advantage** of Spring 7 is that you can switch strategies **centrally without touching controllers**.

### Example 1: Path → Query Param

```java
// OLD
configurer.usePathSegment(2);

// NEW
configurer.useQueryParam("version");
```

---

### Example 2: Query Param → Header

```java
// OLD
configurer.useQueryParam("version");

// NEW
configurer.useRequestHeader("X-API-Version");
```

---

### Example 3: Header → Media Type

```java
// OLD
configurer.useRequestHeader("X-API-Version");

// NEW
configurer.useMediaTypeParameter("v");
```

---

👉 **Controllers remain unchanged:**

```java
@GetMapping(value = "/hello", version = "1.0")
```

✔ No refactoring needed  
✔ Strategy is completely decoupled

## 🧩 Version-Based Request Mapping (NEW)

```java
@RestController
@RequestMapping("/api")
public class VersionController {

    @GetMapping(value = "/hello", version = "1.0")
    public String v1() {
        return "Hello from V1";
    }

    @GetMapping(value = "/hello", version = "2.0")
    public String v2() {
        return "Hello from V2";
    }
}
```

---

## 🔄 How It Works

Spring matches:
- URL path
- HTTP method
- Version (from configured source)

👉 Correct handler method is selected

## 🔁 Supported Versions

```java
configurer.addSupportedVersions("1.0", "2.0");
```

- Only listed versions are valid  
- Unsupported versions → 404 / error  

---

## ⚠️ Important Rules

- Version must exist in configured location  
- Only one strategy should be active  
- Controllers must align with base path  
- `version` attribute requires config  

---

## ❌ Common Mistakes

### Missing Configuration
```java
@GetMapping(value = "/hello", version = "1.0") // ❌ Won’t work
```

### Multiple Strategies at Once
```java
configurer.useQueryParam("version")
          .useRequestHeader("X-API-Version"); // ❌ Avoid confusion
```

### Missing Version
```
/api/hello → ❌ (unless default is set)
```

---

## ✅ Best Practices

- Use **Path versioning** for public APIs  
- Use **Header/Media type** for internal APIs  
- Keep version format consistent (1.0, 2.0)  
- Always define supported versions  
- Use default version for backward compatibility  
- Avoid breaking changes within same version  
- Deprecate older versions gradually  

---

## 🚀 Summary

- Spring 7 introduces native API versioning  
- Supports multiple strategies:
  - Path (`usePathSegment`)
  - Query (`useQueryParam`)
  - Header (`useRequestHeader`)
  - Media Type (`useMediaTypeParameter`)
- `setDefaultVersion()` provides fallback  
- Strategy can be changed **centrally with zero controller changes**  
- Clean, flexible, and production-ready versioning system
