# API Versioning

API Versioning is the practice of **managing changes to an API without breaking existing clients**.  
When an API evolves (new fields, removed fields, changed behavior), versioning allows **older clients to continue working while new clients use updated features**.

---

# Why API Versioning is Important

1. **Backward Compatibility**  
   Existing applications using the API should continue working.

2. **Safe Evolution**  
   APIs can improve without forcing all clients to update immediately.

3. **Multiple Client Support**  
   Different applications may use different versions simultaneously.

4. **Controlled Deprecation**  
   Old versions can be deprecated gradually.

---

# Example Problem Without Versioning

## Initial API

```
GET /api/users
```

Response

```json
{
  "id": 1,
  "name": "John"
}
```

## Later Change

If the API changes to:

```json
{
  "userId": 1,
  "fullName": "John"
}
```

Older applications expecting `id` and `name` will **break**.

Versioning prevents this.

---

# API Versioning Strategies

## 1. URI Versioning (Most Common)

Version is included in the **URL path**.

Example

```
/api/v1/users
/api/v2/users
```

### Spring Boot Example

```java
@RestController
@RequestMapping("/api/v1")
public class UserControllerV1 {

    @GetMapping("/users")
    public String getUsers() {
        return "Users API Version 1";
    }
}
```

Another version

```java
@RestController
@RequestMapping("/api/v2")
public class UserControllerV2 {

    @GetMapping("/users")
    public String getUsers() {
        return "Users API Version 2";
    }
}
```

Request examples

```
GET /api/v1/users
GET /api/v2/users
```

---

## 2. Request Parameter Versioning

Version is passed as a **query parameter**.

Example

```
/api/users?version=1
/api/users?version=2
```

### Spring Boot Example

```java
@GetMapping(value="/users", params="version=1")
public String getUsersV1() {
    return "Users API Version 1";
}

@GetMapping(value="/users", params="version=2")
public String getUsersV2() {
    return "Users API Version 2";
}
```

Request examples

```
GET /api/users?version=1
GET /api/users?version=2
```

---

## 3. Header Versioning

Version is passed in the **request header**.

Example Header

```
X-API-VERSION: 1
```

### Spring Boot Example

```java
@GetMapping(value="/users", headers="X-API-VERSION=1")
public String getUsersV1() {
    return "Users API Version 1";
}

@GetMapping(value="/users", headers="X-API-VERSION=2")
public String getUsersV2() {
    return "Users API Version 2";
}
```

Request example

```
GET /api/users
X-API-VERSION: 1
```

---

## 4. Media Type Versioning (Content Negotiation)

Version is passed inside the **Accept header**.

Example

```
Accept: application/vnd.company.app-v1+json
```

### Spring Boot Example

```java
@GetMapping(value="/users", produces="application/vnd.app-v1+json")
public String getUsersV1() {
    return "Users API Version 1";
}

@GetMapping(value="/users", produces="application/vnd.app-v2+json")
public String getUsersV2() {
    return "Users API Version 2";
}
```

Request example

```
GET /api/users
Accept: application/vnd.app-v1+json
```

---

# Comparison of Versioning Methods

| Method | Example | Pros | Cons |
|------|------|------|------|
| URI Versioning | `/api/v1/users` | Simple, widely used | URL changes |
| Query Param | `/api/users?version=1` | Easy to test | Less RESTful |
| Header Versioning | `X-API-VERSION:1` | Clean URLs | Harder to test |
| Media Type | `Accept: v1` | Most RESTful | Complex |

---

# Best Practice

Most companies use **URI Versioning**.

Example

```
/api/v1/orders
/api/v2/orders
```

Because it is:

- Simple
- Clear
- Easy to document
- Easy to test

# Version Lifecycle

Typical API lifecycle

```
v1 → stable
v2 → new features
v1 → deprecated
v1 → removed
```

Example deprecation header

```
Deprecation: true
Sunset: 2026-12-31
```
