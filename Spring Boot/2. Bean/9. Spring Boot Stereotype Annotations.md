# Spring Boot Stereotype Annotations: Key Differences and Use Cases

In Spring Boot, the annotations `@Component`, `@Service`, `@Repository`, and `@Controller` are **stereotype annotations** used for **automatic bean detection through component scanning**.

All of them are internally derived from `@Component`, but each represents a **different layer in the application architecture**.

---

# 1. Purpose Difference

| Annotation | Layer | Main Purpose |
|---|---|---|
| `@Component` | Generic | General-purpose Spring bean |
| `@Service` | Service Layer | Business logic implementation |
| `@Repository` | Data Access Layer | Database interaction |
| `@Controller` | Presentation Layer | Handles HTTP requests |

---

# 2. Architectural Role

A typical Spring Boot application follows a **layered architecture**:

```
Client Request
      ↓
Controller Layer
      ↓
Service Layer
      ↓
Repository Layer
      ↓
Database
```

Each annotation is intended for **one specific layer**.

| Layer | Annotation | Responsibility |
|---|---|---|
| Presentation | `@Controller` | Handles incoming HTTP requests |
| Business Logic | `@Service` | Processes application logic |
| Data Access | `@Repository` | Communicates with the database |
| Utility / Generic | `@Component` | Used when no specific layer fits |

---

# 3. Special Features

| Annotation | Special Feature |
|---|---|
| `@Component` | Basic bean registration |
| `@Service` | Indicates business logic (semantic clarity) |
| `@Repository` | Automatic database exception translation |
| `@Controller` | Used by Spring MVC to process web requests |

---

# 4. Use Case Examples

## 4.1 @Component – Generic Bean

Used when the class **does not belong to service, repository, or controller layers**.

### Example Use Cases

- Utility classes  
- Email helpers  
- File processors  

### Example

```java
@Component
public class EmailUtility {

    public void sendEmail(String email) {
        System.out.println("Sending email to " + email);
    }
}
```

---

## 4.2 @Service – Business Logic Layer

Used when the class contains **business rules or core application logic**.

### Example Use Cases

- Payment processing  
- Order processing  
- Authentication logic  

### Example

```java
@Service
public class PaymentService {

    public void processPayment() {
        System.out.println("Processing payment...");
    }
}
```

**Best Practice**

Controllers should **not directly access repositories**.  
They should communicate through the **service layer**.

---

## 4.3 @Repository – Database Layer

Used for **database operations**.

### Example Use Cases

- CRUD operations  
- JPA repositories  
- DAO classes  

### Example

```java
@Repository
public class UserRepository {

    public String findUser() {
        return "User from database";
    }
}
```

### Important Feature

Spring automatically converts database exceptions like:

```
SQLException
```

into Spring’s unified exception:

```
DataAccessException
```

This is called **Persistence Exception Translation**.

---

## 4.4 @Controller – Web Layer

Used to **handle HTTP requests**.

### Example Use Cases

- REST endpoints  
- Web page controllers  
- API endpoints  

### Example

```java
@Controller
public class HomeController {

    @GetMapping("/home")
    public String home() {
        return "home";
    }
}
```

This returns a **view** such as:

- HTML
- JSP
- Thymeleaf

---

# 5. Real Spring Boot Example

## Controller

```java
@RestController
public class UserController {

    @Autowired
    private UserService service;

    @GetMapping("/users")
    public String getUsers() {
        return service.getUsers();
    }
}
```

---

## Service

```java
@Service
public class UserService {

    @Autowired
    private UserRepository repository;

    public String getUsers() {
        return repository.fetchUsers();
    }
}
```

---

## Repository

```java
@Repository
public class UserRepository {

    public String fetchUsers() {
        return "Users fetched from DB";
    }
}
```

### Application Flow

```
Client → Controller → Service → Repository → Database
```

---

# 6. Important Interview Concept

Technically, all these annotations behave like `@Component`.

Because internally:

```
@Service
@Repository
@Controller
        ↓
     @Component
```

However, we use them to maintain **clean architecture and readability**.

---

# 7. When to Use Which

| Scenario | Annotation |
|---|---|
| Utility/helper class | `@Component` |
| Business logic | `@Service` |
| Database interaction | `@Repository` |
| Handling HTTP requests | `@Controller` |

---

# 8. Simple Rule to Remember

```
Controller → Service → Repository
```

Each layer has its own stereotype annotation.
